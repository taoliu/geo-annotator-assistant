# Milestone v0.6 — RAG & Validation Robustness

**Status:** CLOSED
**Date:** 2026-01

---

## 1. Milestone Intent

Milestone v0.6 focused on strengthening the **correctness, determinism, and performance** of the backend annotation pipeline, with particular emphasis on:

* ontology grounding reliability,
* retrieval determinism,
* validation semantics,
* and performance stability at GSE scale.

This milestone explicitly **did not change** the core pipeline architecture, output schema, or curator UI semantics.

---

## 2. Scope and Constraints

### In scope

* Ontology grounding robustness
* Deterministic-first RAG retrieval
* Validation and repair correctness
* Performance optimizations that preserve behavior
* Configuration clarity for retrieval and embedding

### Out of scope

* Any redesign of the backend pipeline
* Any change to the 8-field output schema
* Persistence, learning, or stateful behavior
* Curator UI redesign (deferred)

---

## 3. Key Changes Delivered

### 3.1 Deterministic-first ontology retrieval

Ontology retrieval was reworked to prioritize **deterministic exact matches** using normalized metadata before any approximate or vector-based search.

Effects:

* Canonical ontology terms are reliably retrieved
* Approximate nearest-neighbor failures are eliminated
* Vector search is used only as fallback

This aligns retrieval behavior with architectural invariants defined in the whitepaper.

---

### 3.2 Terminal exact match semantics

The concept of a **terminal exact match** was formalized:

* `status == MATCHED`
* `score == 1.0`
* exact match types only

When a terminal exact match is identified:

* Retrieval short-circuits
* No embedding calls are performed
* Vector fallback is skipped

This change preserves behavior while significantly improving performance.

---

### 3.3 Canonicalization and locking of exact matches

For terminal exact ontology matches:

* Output values may be deterministically canonicalized to ontology labels
* Fields may be locked to prevent later repair overwrites

Both behaviors are:

* configuration-gated,
* deterministic,
* fully auditable.

This reduces downstream repair churn and improves output consistency.

---

### 3.4 Disease ontology fallback (DOID → NCIT)

Disease grounding was extended to support a **controlled secondary ontology**:

* Human Disease Ontology (DOID) remains primary
* NCIT is queried only when:

  * DOID is not terminal exact, and
  * the disease label matches a malignant-neoplasm lexical trigger

The trigger lexicon is configurable, deterministic, and transparent.

This enables accurate grounding of detailed cancer subtypes without over-triggering.

---

### 3.5 Performance and embedding control

Additional improvements:

* Embedding calls are avoided whenever possible
* Vector search is strictly fallback-only
* Embedding device (`cpu`, `cuda`, `mps`) is explicitly configurable

These changes resulted in a substantial end-to-end speedup without altering semantics.

---

## 4. What Did Not Change

The following architectural properties remain unchanged:

* Exact 8-field output schema
* Pipeline ordering and decision routing
* Repair loop structure and bounds
* RAG read-only, non-decisional role
* No learning or persistence

Milestone v0.6 is a **robustness and clarity milestone**, not a redesign.

---

## 5. Outcome and Impact

Milestone v0.6 achieved:

* Higher ontology grounding accuracy
* Dramatically improved performance
* Fewer spurious repair attempts
* Clearer audit traces
* Stronger guarantees for future UI work

The backend is now considered **stable and performant** for further UI-focused development.

---

## 6. Next Milestone

The next planned milestone returns focus to **curator UI design and usability**:

* improved visualization
* clearer grounding evidence presentation
* safer and more efficient override workflows

Backend semantics introduced in v0.6 are considered **locked** unless explicitly revised by future tickets.

## 7. Milestone Closure

Milestone v0.6 is complete and closed.

All changes were implemented via ticketed, incremental work and are consistent with the long-term architectural invariants defined in `docs/whitepaper.md`.
