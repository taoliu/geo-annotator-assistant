# Milestone v1.2 — CLI Ergonomics & Agent Invocation Refinement

## Status

**Completed**

## Scope

Milestone v1.2 focused exclusively on **command-line interface (CLI) ergonomics, invocation behavior, ingest robustness, and post-processing workflows** for the GEO GSM Annotator Agent.

This milestone intentionally **did not** modify:
- validation logic
- repair logic
- ontology grounding semantics
- decision routing
- output schema
- curator UI behavior

All backend semantics defined in v0.9 and earlier remain authoritative and unchanged.

---

## Motivation

With the backend stabilized and curator UI v1.1 in active use, real-world usage exposed several friction points:

- ambiguity in CLI batch invocation
- inefficient reuse of expensive runtime resources on clusters
- fragile ingest behavior when local GEO SOFT repositories contain incomplete files
- lack of visibility into long-running CLI jobs
- absence of a clean, CLI-native post-curation summarization workflow

Milestone v1.2 addresses these issues while preserving determinism, auditability, and policy governance.

---

## Completed Work

### 1. Multi-GSE CLI support

- Added explicit support for processing multiple GSEs from a plain-text file.
- Batch behavior is explicit and never inferred.
- Output layout is deterministic and auditable.

### 2. LLM lifecycle reuse (local transformers only)

- Ensured that LLM models using `local_transformers` are instantiated once per CLI invocation.
- Reuse is gated by transport type to avoid unintended coupling for remote or HTTP-based backends.
- No changes to prompts, generation parameters, or repair behavior.

### 3. Robust local-first GEO SOFT ingest with remote fallback

- Introduced local-first resolution of GEO SOFT files from a configured repository.
- Added configurable remote fallback (FTP or HTTPS).
- Implemented retry-on-failure semantics:
  - missing local file triggers remote download
  - corrupted or unparsable local file triggers forced re-download and retry
- Repository self-heals by overwriting incomplete files with fresh downloads.

### 4. Configurable remote transport for GEO SOFT download

- Users may select FTP or HTTPS for remote SOFT downloads via configuration.
- Behavior is explicit, deterministic, and validated.

### 5. Runtime observability via `--verbose`

- Added a `--verbose` flag to emit step-level runtime messages.
- Messages cover:
  - SOFT resolution and parsing
  - LLM invocation
  - validation and ontology grounding
  - decision routing and repair
  - output emission
- Verbose output is informational only and does not alter audit artifacts.

### 6. Chroma client and collection reuse

- Ensured Chroma client and ontology collections are constructed once per process invocation.
- Eliminated repeated index opening and first-query penalties across GSMs.
- Preserves read-only semantics and determinism.
- Significantly improves performance on cluster environments with shared filesystems.

### 7. Standalone post-curation summarization command

- Introduced a new CLI entry point: `geo-gsm-summarize`.
- Reads existing output directories without invoking the agent.
- Applies curator overrides.
- Exports:
  - GSM-level CSV with 8 canonical fields.
  - GSE-level CSV with 7 canonical fields.
- Reuses the same summarization logic as the curator UI.

---

## Explicit Non-Goals (Respected)

- No schema changes.
- No reinterpretation of policy semantics.
- No automatic learning from curator edits.
- No UI changes.
- No parallel execution changes.
- No ontology or RAG behavior changes beyond handle reuse.

---

## Performance Characteristics

After v1.2:
- Expensive resources (LLMs, Chroma collections) are reused per invocation.
- First-use penalties are amortized across batch runs.
- The system behaves predictably on cluster GPU nodes when combined with node-local storage.

---

## Known Follow-Ups (Deferred)

The following were identified but intentionally deferred:

- README.md refresh (to be handled by a code-aware agent).
- Removal of `--overrides` from `geo-gsm-annotate` to further enforce workflow separation.
- Optional ontology warmup hooks.
- Environment-variable expansion in config paths.

These are candidates for future milestones but are not required for correctness.

---

## Summary

Milestone v1.2 completes the transition from a functional prototype CLI to a **production-grade, batch-safe, cluster-friendly command-line interface**, while preserving the architectural and policy invariants of the system.

The agent is now:
- easier to invoke correctly,
- faster in real-world batch settings,
- more observable,
- and cleanly separated from post-curation reporting workflows.
