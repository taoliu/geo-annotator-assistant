# Milestone v0.1 — Local LLM Annotation Pipeline

**Date:** 2026-01-03  
**Tag:** `geo-gsm-annotator-agent-v0.1-llm`

## Summary

This milestone marks the first **fully working end-to-end GEO GSM annotation pipeline** using a **local in-process LLM** (HuggingFace Transformers), with robust validation, repair, auditing, and batch GSE support.

The system can now:
- Take a GSE accession (via pre-built JSONL contexts)
- Annotate each GSM using a local instruction-tuned LLM
- Enforce strict output format and constraints
- Repair invalid or noisy model outputs automatically
- Produce reproducible annotation, audit, and flagged outputs

This milestone transitions the project from a prototype to a stable, extensible foundation.

---

## What Works

### 1. Input and Ingestion
- GEO SOFT files can be downloaded and parsed externally.
- Each GSM is converted into a JSONL record with:
  ```json
  {
    "context_text": "... GEO metadata ...",
    "gsm_accession": "GSMxxxx",
    "gse_accession": "GSExxxx"
  }
  ```

* Ingestion is cleanly separated from agent logic.

### 2. Prompting

* Instruction prompt lives in `prompts/label_v1.txt`.
* Agent assembles:

  ```
  label prompt + context_text
  ```
* Chat templates are applied by default via tokenizer.

### 3. Local LLM Integration

* Uses HuggingFace Transformers in-process (no HTTP server required).
* Tested with:

  * `llama-3.2-1b-it` (local directory, safetensors)
* Supports:

  * stub mode (for testing)
  * local transformers mode
* Configuration-driven via YAML.

### 4. Robust Format Handling

* Model outputs may include:

  * Markdown code fences
  * Extra explanations
  * Partial or noisy text
* Format validator now:

  * Extracts the first valid JSON object from raw text
  * Handles fenced JSON blocks
* Prevents false failures due to harmless verbosity.

### 5. Automatic Repair Loops

Two repair mechanisms are implemented:

#### a) Format Repair (Pre-loop)

* Triggered when format validation fails.
* Re-prompts the LLM using `repair_format_v1.txt`.
* Stops at the **first successful parse**.
* Prevents later failed attempts from overwriting success.

#### b) Decision-Based Repair

* Consistency and semantic failures are routed via a decision table.
* REPAIR actions call the LLM again (full 8-field JSON).
* FALLBACK values are applied when repair attempts are exhausted.

### 6. Consistency Flag Handling

* Consistency flags (e.g. `healthy_disease_conflict`) are **not auto-fatal**.
* They are routed through the decision engine.
* High-severity unresolved conflicts can still escalate to FLAGGED.

### 7. Batch GSE Mode

* JSONL-based GSE batch processing is supported.
* CLI:

  ```bash
  agent.cli --jsonl GSE112494.jsonl --config config.yaml
  ```
* Outputs:

  * `annotations.jsonl`
  * `audit.jsonl`
  * `flagged.jsonl`
* All GSMs processed independently with isolated audit trails.

### 8. Audit and Reproducibility

Each GSM produces a detailed audit record including:

* Prompt and validator versions
* All raw LLM outputs (initial + repairs)
* Parsed outputs
* Validation errors and consistency flags
* Repair history
* Final decision and output
* Timestamp

This makes the pipeline fully debuggable and reproducible.

## Known Limitations (Expected)

* The 1B model has limited semantic understanding:

  * Often confuses tissue vs cell type
  * May mislabel assay vs molecule
* Ontology grounding is not yet implemented.
* No ensemble or escalation to larger models yet.
* `--gse` one-command workflow (download + parse + annotate) is not fully integrated inside the agent.

These are design limitations, not bugs.

---

## Key Design Decisions (Do Not Undo Lightly)

1. **Ingest vs Agent separation**

   * Ingest produces context only.
   * Agent owns prompting and reasoning.

2. **Instruction prompt versioning**

   * Prompts live in `prompts/`
   * Prompt text is part of the contract.

3. **Repair uses full JSON**

   * Repairs always return the full 8-field schema.
   * Simplifies merging and validation.

4. **Format robustness over strictness**

   * JSON extraction is allowed.
   * Structural correctness is enforced, not stylistic purity.

5. **Consistency flags are not fatal by default**

   * They inform repair and escalation, not automatic rejection.
