# Milestone v0.7 — Curator UI Refinement

**Status:** CLOSED
**Date:** 2026-01

---

## 1. Milestone Intent

Milestone v0.7 focused exclusively on **curator-facing UI usability and clarity**, building on a backend that was declared stable and frozen after v0.6.

The goal was to make backend decisions:

* visible,
* understandable,
* safely overridable,

without changing **any backend semantics, schema, or decision logic**.

This milestone is intentionally UI-only.

---

## 2. Scope and Constraints

### In scope

* Curator UI interaction redesign
* Visualization of ontology grounding results
* Clear signaling of:

  * terminal exact matches
  * canonicalized fields
  * locked fields
  * ambiguous or no-match cases
* Evidence presentation based on existing audit artifacts
* Safer, clearer override workflows (session-only)
* Table-level ergonomics for large GSEs

### Out of scope

* Backend redesign
* Ontology logic changes
* Validation or repair logic changes
* Persistence or saving UI state
* Learning from curator edits
* Any schema changes

All backend semantics introduced in v0.6 are treated as **frozen**.

---

## 3. Major UI Changes Delivered

### 3.1 Modal-based GSM inspection

* GSM details are no longer rendered below the curation table.
* Clicking a GSM in the table opens a **modal popup** showing record details.
* The table remains the primary navigation and comparison surface.

This eliminates scroll-heavy workflows and preserves context.

---

### 3.2 GSM accession as primary interaction affordance

* The `gsm_accession` column is visually emphasized as the entry point.
* Inline guidance clarifies how to open record details.
* Active rows are highlighted to reinforce table–modal linkage.

---

### 3.3 Field Status Dashboard

A compact dashboard at the top of the modal summarizes the state of all 8 fields:

* backend final value
* terminal exact / canonicalized / locked signals
* ambiguous or no-match status
* session overrides

This enables at-a-glance assessment before drilling into details.

---

### 3.4 Structured evidence panels

For ontology-backed fields (`data_type`, `tissue_type`, `cell_line`, `disease`):

* Expandable evidence panels display structured facts only:

  * raw value
  * normalized value
  * ontology source
  * match type
  * score
  * canonical label used
* No free-text rationale is generated.

Evidence is read-only and derived solely from existing audit artifacts.

---

### 3.5 Override ergonomics with soft safety

* All fields remain editable when editing is enabled.
* High-confidence backend signals (locked, canonicalized, terminal exact) trigger:

  * clear inline warnings
  * explicit confirmation before applying overrides
* Overridden fields display backend vs override diffs.
* Per-field revert is supported.

Overrides remain session-only and do not affect backend logic.

---

### 3.6 Table-level triage and filters

* Summary counts above the table show:

  * total GSMs
  * GSMs needing attention
  * GSMs with overrides
  * clean GSMs
* Quick UI-only filters support rapid triage:

  * Needs attention
  * Has overrides
  * Clean
  * All

Filtering affects display only and never alters data or exports.

---

## 4. What Did Not Change

The following properties remain unchanged and were explicitly protected:

* Exact 8-field output schema
* Backend pipeline ordering
* Deterministic decision routing
* Ontology grounding semantics
* Repair loop structure and bounds
* RAG read-only, fallback-only role
* No persistence or learning

---

## 5. Outcome and Impact

Milestone v0.7 achieved:

* Significantly improved curator efficiency
* Reduced cognitive load when reviewing large GSEs
* Clear separation between backend truth and human judgment
* Safer override workflows without restricting curator authority

The UI now faithfully exposes backend decisions without reinterpreting them.

---

## 6. Next Milestone

The next milestone will return focus to the **end-to-end pipeline**, with goals to:

* re-walk the entire processing flow using real datasets
* identify edge cases and inconsistencies
* fix issues incrementally via small, targeted tickets

This work will proceed **without altering architectural invariants**.

## 7. Milestone Closure

Milestone v0.7 is complete and closed.

All changes were:

* UI-only
* ticketed
* incremental
* consistent with the long-term architectural law defined in `docs/whitepaper.md`.
